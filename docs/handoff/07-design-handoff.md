# Chappy Web v0.1 — Дизайн-система и хендофф

Документ для разработчика, который будет собирать продакшн-версию. Описывает дизайн-систему прототипа как она **реально** реализована в коде, а не как задумывалась. Все токены, классы и правила взяты из исходников.

## Где живёт дизайн-система

Вся система — это три глобальных CSS-файла плюс несколько CSS Modules для лейаутов. Никакого Tailwind, styled-components или CSS-in-JS рантайма нет; компоненты используют глобальные утилити-классы и точечные inline-стили.

| Файл | Роль |
|---|---|
| `/Users/nutls/Downloads/chappy-web-prototype-v0.1/src/styles/tokens.css` | Все дизайн-токены в `:root` (цвета, радиусы, спейсинг, шрифты, лейаут, motion) + `@import` Google Fonts |
| `/Users/nutls/Downloads/chappy-web-prototype-v0.1/src/styles/global.css` | Reset, `box-sizing`, базовая типографика (`h1–h4`), фокус, скроллбары, `prefers-reduced-motion`, утилиты (`.wrap`, `.eyebrow`, `.grad-text`, `.muted`, `.mono`) |
| `/Users/nutls/Downloads/chappy-web-prototype-v0.1/src/styles/components.css` | Токен-driven компонентные классы (`.btn`, `.card`, `.badge`, `.chip`, `.input`, грид-хелперы, типографика `.h1/.h2/.h3/.sub/.k`) |
| `src/layouts/*.module.css`, `src/features/canvas/canvas.css` | Локальные стили оболочек и Canvas (React Flow) |

Порядок подключения задан в `src/main.tsx`: `tokens.css` → `global.css` → `components.css`. Тему определяет только `:root` — **светлой темы нет**, приложение всегда тёмное.

### `--scrim`

`--scrim` (`rgba(5,6,12,.7)`) задуман как затемняющий слой под модалками/оверлеями. **Важно:** в текущем коде токен объявлен, но оверлеи (мобильный drawer в `AppLayout.module.css`, подложка `Modal`) используют собственные литералы `rgba(3,4,10,.6)` / `rgba(5,6,12,.82)`, а не `var(--scrim)`. При переносе стоит свести всё к одному токену.

## UX-правила прототипа

Прототип последовательно проводит «честную» линию: ничего не притворяется работающим, если не работает.

- **Честные «будущие» страницы** — `src/components/FuturePage.tsx`. Для скоупа вне v0.1: бейдж «В следующих версиях», описание что будет, список фич и **всегда живой выход** в работающую область (никаких тупиков). Используется для всех отложенных публичных и воркспейс-роутов.
- **Индикатор демо-режима** — `src/components/DemoBanner.tsx`. Ненавязчивая фиксированная пилюля слева-снизу (зелёная точка, `--grad-soft`), по клику запускает `DemoTour`.
- **Мок-бейджи** — `src/components/MockBadge.tsx`. Золотой `.badge-gold` с точкой и тултипом (`ru.mockNote`) помечает демо-только функциональность.
- **Гейты человеческого одобрения** — прогоны команд агентов имеют статус `waiting_approval` (`src/types/index.ts`, `src/features/agents/runStatus.ts`). Команда создаётся с флагом `approvalRequired` (`TeamBuilderModal`, дефолт включён), запуск показывает «human approval: да/нет» (`RunTeamModal`), а `approveRun()` в `src/features/agents/orchestration.ts` фиксирует одобрение, создаёт ассет и контрольную точку. Пользователь может одобрить / отправить на доработку / остановить.
- **Подтверждение деструктивных действий** — через `window.confirm(...)` перед необратимым: очистка Canvas (`CanvasEditor.tsx:308`), восстановление из контрольной точки (`ProjectCheckpoints.tsx:29`), удаление записи памяти проекта (`ProjectMemory.tsx:57`).

Эти правила стоит сохранить в проде как продуктовые инварианты, заменив мок-механику на реальные API/статусы.

## Известный визуальный долг

Список для команды продакшна — что переделать при переносе дизайн-системы:

1. **`.sub` почти всегда переопределяется инлайн.** Класс задаёт `font-size:16px`, но из 75 использований 62 (~83%) сразу же перекрывают размер инлайном (`style={{ fontSize: 12/13/13.5/14 }}`). Значение 16px де-факто мёртвое. Нужны семантические варианты (`.sub-sm`, `.sub-xs`) или размерный проп, а не литералы у каждого элемента.
2. **Ad-hoc литералы спейсинга и радиусов.** 8px-шкала (`--s1…--s8`) объявлена, но компоненты и грид-хелперы задают отступы/радиусы числами (`gap:16`, `padding:20px`, `borderRadius: 12/10/8/14`, `marginBottom: 8/10/20` и т.п. по всему `src/components` и `src/pages`). Радиусы-литералы не совпадают со шкалой токенов (`6`, `10`, `14` не равны `--r-sm/--r/--r-lg`). Нужна ревизия под токены.
3. **Google Fonts через `@import`.** Шрифты тянутся `@import url('https://fonts.googleapis.com/...')` в начале `tokens.css` — это рендер-блокирующий запрос к внешнему CDN, конфликтует с CSP/офлайном/приватностью и добавляет FOUT. В проде — self-host (woff2 + `@font-face` + `preload`) или хотя бы `<link rel="preconnect">`.
4. **Дублирование `.card`.** Идентичное правило объявлено и в `global.css`, и в `components.css`. Оставить одно.
5. **`--scrim` объявлен, но не используется.** Оверлеи задают собственные `rgba(...)` литералы вместо `var(--scrim)` (`AppLayout.module.css`, `Modal`). Свести к токену.
6. **Дубли статусных цветов и отсутствие семантических пар.** `--danger === --hot`. Border-цвета бейджей захардкожены как `rgba(...)` внутри `.badge-*` вместо производных от статус-токенов. Нет фон/бордер-вариантов статусов — при расширении палитры придётся заводить систему.
7. **`.k` vs `.eyebrow` — перекрывающиеся роли.** Два почти одинаковых mono-надзаголовка (отличаются цветом и tracking). Стоит объединить в один компонент с вариантом «акцентный».
8. **Только тёмная тема.** Все токены в `:root`, светлого набора нет; переключение темы потребует рефакторинга в `data-theme`/`prefers-color-scheme`.
9. **Магические числа в UI-логике.** Прогресс баланса считается как `credits/1200` (`AppLayout.tsx`), оценка стоимости/времени прогона — `memberIds.length * 2` (`RunTeamModal`); при переезде на реальные данные это надо вынести в конфиг/API.
10. **Смешанная стилевая стратегия.** CSS Modules используются только для двух лейаутов и Canvas; всё остальное — глобальные утилити-классы + большие блоки инлайн-стилей в JSX. Единого подхода нет — для продакшна лучше выбрать одну модель (напр. CSS Modules или токен-driven утилиты) и придерживаться её.
