# 08 · Demo Script — Chappy Web v0.1 (5–7 минут)

Сценарий живой демонстрации прототипа. Всё показывается на seed-данных в браузере — бэкенда нет, генерация/оплата/агенты имитируются и честно помечены. Перед показом (по желанию) откройте **Профиль → «Восстановить демонстрационные данные»**, чтобы состояние было исходным.

> Метка **[MOCK]** = имитация прототипа. Метка **[REAL]** = реально работающая логика фронтенда (состояние, связи, персист).

---

## 0. Подготовка (10 сек, до показа)
- **Нажать:** открыть `/` (главная Chappy Web).
- **Показать:** заголовок «Рабочее пространство для AI-контента».
- Плашка **«Демонстрационный режим»** внизу слева — можно нажать, чтобы в любой момент запустить встроенный тур (8 шагов).

---

## 1. Главная Chappy Web (30 сек)
- **Нажать:** ничего, прокрутить hero → блок «Главное отличие Chappy» → «Проекты» → «Команды агентов».
- **Показать:** hero (проекты, Canvas, аватары, команды), секция Canvas с мини-схемой процесса, секция Проектов, пайплайн команды агентов.
- **Сказать:** «Chappy Web — это рабочее пространство для AI-контента. Не список моделей, а среда, где весь процесс живёт в проектах и собирается на бесконечной доске».
- **Ценность:** за 10 секунд видно, что это workspace, а не агрегатор.
- **[REAL]** статичный лендинг.

## 2. Отличие от обычного агрегатора моделей (20 сек)
- **Показать:** блок «Бесконечный Canvas — весь процесс на одной доске» и пайплайн «Исследователь → Сценарист → Креативный директор → Prompt Engineer → QA».
- **Сказать:** «Агрегатор — это ‘дай промпт, получи картинку’. Здесь — проекты с памятью, переиспользуемые аватары и команды агентов, которые проходят весь процесс с человеческим одобрением».
- **Ценность:** позиционирование как продукт-процесс, а не разовая генерация.

## 3. Workspace (20 сек)
- **Нажать:** в hero — **«Открыть демо-проект»** (или «В рабочее пространство» → `/app`).
- **Показать:** app-shell — сайдбар (Рабочее пространство / Моё), топбар с балансом кредитов, на главной — виджеты проектов, аватаров, активности агентов, Project Outputs и чекпоинтов.
- **Сказать:** «Это рабочий стол. Слева — разделы, вверху — баланс. Всё, что важно, собрано на одном экране».
- **Ценность:** цельный продукт, а не набор отдельных инструментов.
- **[REAL]** данные из стора; **[MOCK]** баланс кредитов — клиентское число.

## 4. Проект «Кира — запуск AI-продукта» (30 сек)
- **Нажать:** карточку проекта **«Кира — запуск AI-продукта»** (метка «Продолжить») → `/app/projects/pr-kira`.
- **Показать:** заголовок, описание, вкладки **Canvas / Ассеты / Агенты-Запуски / История**, счётчики (4 ассета, 1 запуск).
- **Сказать:** «Открываем готовый проект — не пустой холст, а собранный кейс: от идеи до утверждённого кадра».
- **Ценность:** проект как единица хранения работы.
- **[REAL]** проект и его Canvas загружаются из стора.

## 5. Canvas (40 сек)
- **Нажать:** вкладка **Canvas** (открыта по умолчанию), при необходимости кнопка **Fit**.
- **Показать:** предсобранный граф — нода **Идея** и блоки **Аватар (Кира) / Стиль / Локация** слева соединены рёбрами с нодой **Агент (Контент-команда)** и нодой **AI-модель**, которые ведут к нодам-результатам (**вывод агента** и **модель**).
- **Сказать:** «Весь процесс — на одной доске. Блоки соединяются: идея и сущности подаются в команду агентов и в модель, а справа — результаты».
- **Ценность:** визуальный, повторяемый процесс вместо цепочки отдельных экранов.
- **[REAL]** граф (React Flow) и связи; **[MOCK]** сами картинки — из демо-пула.

## 6. Постоянный аватар, стиль, голос и локация (40 сек)
- **Нажать:** сайдбар **Аватары** → карточка **«Кира»** (или нода «Аватар» на Canvas → «Открыть»); адрес `/app/avatars/av-kira`.
- **Показать:** референсы Киры, панель **«Связанные сущности»**: Голос — «Кира — тёплый женский», Стиль — «Продуктовый неон», Локация — «Светлая студия», Проекты · 1.
- **Сказать:** «Аватар сохраняется один раз со своим голосом, стилем и локацией — и подставляется в любую генерацию и в любой проект».
- **Ценность:** переиспользуемые сущности = консистентность героя между кадрами и проектами.
- **[REAL]** сущности и их связи по стабильным id.

## 7. Команда агентов (30 сек)
- **Нажать:** сайдбар **Агенты** → `/app/agents`.
- **Показать:** карточку команды **«Контент-команда»** (5 агентов, координатор — Креативный директор, approval: да) и карточки ролей с их инструментами.
- **Сказать:** «Команда — это готовый процесс: исследователь, сценарист, директор, prompt-инженер и QA. Её можно запустить на проекте».
- **Ценность:** команда проходит весь путь, а не один агент отвечает на один запрос.
- **[REAL]** модель команд/ролей.

## 8. Human approval (50 сек)
- **Нажать:** на карточке проекта «Кира» кнопку **«Запустить команду»** (или на Canvas в ноде «Агент» — **«Заново»**), в модалке — **«Запустить»**. Откроется `/app/runs/<id>`.
- **Показать:** живой **таймлайн**: этапы `queued → running`, передача между агентами (**→ передано: …**), затем статус **«ждёт одобрения»** и панель **Human approval** (сценарий, аватар/стиль/локация, модель, стоимость, замечания QA) с кнопками **Одобрить / На доработку / Изменить / Остановить**.
- **Нажать:** **«Одобрить»**.
- **Сказать:** «Агенты работают по шагам и передают результат дальше, но перед финалом — обязательное одобрение человеком. Одобряем — и создаётся результат».
- **Ценность:** контролируемый процесс, human-in-the-loop.
- **[MOCK]** прогон на таймерах (`orchestration.ts`), результат — демо-картинка. **[REAL]** состояние прогона, шаги, handoff, gate одобрения, списание mock-стоимости.
- *Примечание:* seed-запуск `run-kira` уже завершён — если хотите показать только результат без ожидания, откройте вкладку **Агенты / Запуски** и существующий прогон.

## 9. Финальный Output (20 сек)
- **Показать:** после одобрения — карточку **«Финальный результат»** с картинкой, `agentRunId` и кнопками **«Открыть в проекте» / «Добавить на Canvas»**. На вкладке **Ассеты** проекта — блок **Project Outputs** (утверждённый результат) с ⭐.
- **Сказать:** «Результат становится ассетом проекта и может быть назначен финальным Output’ом».
- **Ценность:** результат встроен в проект, а не «улетел в чат».
- **[REAL]** ассет со ссылкой на прогон, Project Output.

## 10. Recipe и происхождение Asset (40 сек)
- **Нажать:** сайдбар **Ассеты** → открыть **«Ключевой кадр · оригинал»** (`/app/assets/as-k1`).
- **Показать:** **Generation Recipe** (промпт, модель Flux Pro, аватар Кира, стиль, локация, стоимость) и блок **«Происхождение / Lineage»** — цепочка **оригинал → вариация → апскейл 4K**, связанные сущности, проект, Canvas-нода.
- **Сказать:** «Каждый результат хранит полный рецепт и историю происхождения — из чего он собран и во что превратился».
- **Ценность:** прозрачность и воспроизводимость результата.
- **[REAL]** recipe, lineage-граф, связи по id.

## 11. Вариация (30 сек)
- **Нажать:** на Asset Detail — **«Создать вариацию»**, изменить промпт → **«Запустить»**.
- **Показать:** появляется новый ассет с `derivedFromAssetId`, тип деривации **variation**; в lineage — новая связь.
- **Сказать:** «Из любого результата делаем вариацию — новая версия сохраняет связь с исходником».
- **Ценность:** версии и ответвления, а не хаос из отдельных файлов.
- **[MOCK]** генерация вариации (таймер + демо-картинка). **[REAL]** связь версий.

## 12. История и checkpoint (40 сек)
- **Нажать:** вернуться в проект «Кира» → вкладка **История** (`/app/projects/pr-kira?tab=history`).
- **Показать:** хронологическую ленту (создание проекта → аватар → память → запуск команды → одобрение → создание/деривации ассетов → чекпоинт → Output), фильтры, панель **Контрольные точки** с точкой «После утверждения концепции» и кнопками **Восстановить / Ответвить проект**.
- **Сказать:** «Вся история проекта — по времени. Контрольные точки позволяют вернуться к любому состоянию Canvas или ответвить проект. Перед восстановлением текущее состояние автоматически сохраняется».
- **Ценность:** безопасная работа с версиями проекта, ничего не теряется.
- **[REAL]** activity-лента, чекпоинты, restore/fork через стор.

## 13. Возврат к Workspace (20 сек)
- **Нажать:** сайдбар **Главная** (`/app`).
- **Показать:** обновлённые виджеты (последние ассеты, активность, чекпоинты); при желании — **Профиль → «Восстановить демонстрационные данные»** для сброса перед следующим показом.
- **Сказать:** «Всё это живёт в одном пространстве, переживает перезагрузку и готово к следующему проекту».
- **Ценность:** цельный, постоянный workspace.
- **[REAL]** персист состояния (localStorage), Demo Reset.

---

## Что на всём протяжении — mock (сказать один раз в конце)
- Генерация изображений/видео и работа агентов — **имитация на таймерах**, результаты — из демо-пула картинок.
- Кредиты и оплата (ЮKassa) — **демо**, реального списания нет, данные карты не запрашиваются.
- Авторизации и бэкенда нет — данные хранятся локально в браузере.

## Что реально (главный посыл)
Продуктовая **модель и связи** — проекты, Canvas-граф, переиспользуемые сущности, команды агентов с human approval, происхождение ассетов, история и чекпоинты — реализованы и переживают перезагрузку. Это и есть контракт, который предстоит перенести на реальный бэкенд.
