# 09 · Developer Start Brief — Chappy Web (production, шаг 1)

Короткий бриф «с чего начать», чтобы за первую неделю получить одну **сквозную реально работающую цепочку** вместо широкого, но нерабочего каркаса. Контекст модели данных и контрактов — в [03-domain-model.md](03-domain-model.md) и [04-backend-contracts.md](04-backend-contracts.md); что mock и что переиспользовать — в [06-prototype-vs-production.md](06-prototype-vs-production.md); приоритеты — в [05-production-priorities.md](05-production-priorities.md).

---

## Первая production-цель

**Одна реальная генерация внутри проекта, которая переживает повторный вход.** Не весь продукт — именно эта вертикаль от регистрации до восстановления проекта. Всё остальное (агенты, lineage, чекпоинты, каталог моделей, Music Studio) — после того, как эта цепочка работает на реальном бэкенде.

### Целевая цепочка (Definition: «шаг 1 готов»)

```
Регистрация
 → создать Project
 → открыть Canvas
 → выбрать реальную модель
 → запустить backend generation job
 → получить результат
 → сохранить файл в object storage
 → создать Asset
 → списать кредиты
 → восстановить Project после повторного входа
```

## P0 scope (только необходимое для цепочки)

- **Auth** — регистрация/вход (email или OAuth), сессия/JWT, привязка всех сущностей к `userId`.
- **Backend + DB** — сервис и схема для `users`, `projects`, `canvas_graphs`, `assets`, `generation_jobs`, `credit_ledger` (нормализованно, не «фронтенд-типы как есть»).
- **Object storage** — S3-совместимое хранилище для медиа + выдача подписанных URL.
- **Реальная интеграция модели** — хотя бы одна image-модель (например Flux) через провайдера.
- **Generation jobs** — асинхронно: `create → running → succeeded/failed`, поллинг или стрим статуса; идемпотентность; логика ошибок и **возврата кредитов** при fail.
- **Credits ledger** — серверный баланс и проводки (списание при старте job, возврат при ошибке), не клиентское число.
- **Projects + Canvas persistence** — сохранение/загрузка графа Canvas на сервере.
- **Assets** — запись результата с `recipe`, `projectId`, `modelId`, ссылкой на файл в storage.

Вне P0 (не трогать на шаге 1): агенты/команды, lineage-деривации, чекпоинты/fork, каталог моделей и страницы моделей, Music Studio, Telegram, teams/collaboration.

## Последовательность разработки

1. **Auth + БД-скелет** — `users`, сессии; пустые таблицы `projects`/`assets`/`generation_jobs`/`credit_ledger`.
2. **Projects CRUD + Canvas persistence** — сохранить/загрузить граф (контракт `canvas` из doc 04). Фронт уже умеет сериализовать `CanvasGraph`.
3. **Object storage** — загрузка/выдача файлов, подписанные URL.
4. **Одна модель + generation job** — endpoint `create job` → провайдер → статус → результат в storage → `Asset`.
5. **Credits ledger** — списание на старте job, возврат на fail; баланс из ledger.
6. **Реинтеграция фронта** — заменить mock-слой на реальные вызовы (см. «переиспользовать»), проверить восстановление проекта после повторного входа.

## Что уже существует в Chappy Bot (уточнить у команды бота)

Chappy Bot — **отдельный продукт** (Telegram-бот, «один запрос → один результат»). Кодовой базы бота в этом репозитории нет, поэтому ниже — **кандидаты на переиспользование, которые нужно подтвердить** у владельца бота:

- **Интеграции с моделями** — если бот уже вызывает провайдеров генерации, переиспользовать этот слой (ключи, обёртки, обработку ошибок).
- **Кредитная логика / тарифы** — если у бота есть баланс и списания, взять формулы стоимости и модель ledger.
- **Оплата ЮKassa** — если в боте настроены платежи, переиспользовать интеграцию и вебхуки.
- **Очередь задач генерации** — если бот уже гоняет задачи асинхронно, переиспользовать воркеры/очередь.
- **Аккаунты пользователей** — если есть общая база пользователей, согласовать единый `userId` между Bot и Web.

> Действие на неделе: 30-минутная сверка с командой бота — что из перечисленного существует и переиспользуемо, чтобы не строить с нуля интеграции моделей и платежей.

## Что можно переиспользовать из этого фронтенд-прототипа

- **Модель данных и контракты** — типы `src/types/index.ts` как основа API-контрактов (нормализовать под БД).
- **Весь UI/UX и дизайн-система** — компоненты, токены, экраны Projects/Canvas/Assets готовы; менять только слой данных.
- **Сериализация Canvas** — `CanvasGraph` (nodes/edges с id) уже готов к сохранению/загрузке.
- **Recipe/Asset-структура** — поля рецепта и ассета соответствуют тому, что нужно писать в БД.
- **Swappable-сервисы** — `services/generation.ts` и `features/agents/orchestration.ts` изолируют mock; их API — точки подмены на реальные вызовы.

## Что нельзя переносить напрямую

- **Мок-таймеры** (`services/generation.ts`, `features/agents/orchestration.ts`) → реальные job/stream.
- **Клиентские кредиты** (число в zustand) → серверный ledger.
- **localStorage-как-БД** (zustand persist) → настоящая БД; персист оставить только как локальный UI-кэш.
- **Mock-ЮKassa** → реальная интеграция + вебхуки.
- **Демо-пул картинок в `/public`** → object storage.
- **Отсутствие auth** → обязательный auth + RBAC/ownership по `userId`.
- **Фронтенд-типы как DB-схема** → нормализация, индексы, ограничения, RLS.

## Первые задачи разработчика (неделя 1)

- **Д1:** поднять backend-скелет + БД-миграции (`users`, `projects`, `assets`, `generation_jobs`, `credit_ledger`); auth (регистрация/вход/сессия).
- **Д2:** Projects CRUD + сохранение/загрузка `CanvasGraph`; подключить фронтовые экраны Projects/Canvas к реальным endpoint’ам.
- **Д3:** object storage (bucket, загрузка, подписанные URL); сверка с командой бота по моделям/платежам.
- **Д4:** одна image-модель + `generation_jobs` (create → status → result); запись `Asset` + файл в storage.
- **Д5:** credits ledger (списание/возврат), обработка ошибок job; связать баланс с UI.
- **Буфер:** сквозной прогон цепочки, восстановление проекта после повторного входа, багфикс.

## Критерии готовности первой production-цепочки

1. Новый пользователь **регистрируется** и входит; данные привязаны к его аккаунту.
2. Пользователь **создаёт проект** и открывает **Canvas**; граф сохраняется на сервере.
3. Пользователь **выбирает реальную модель** и запускает генерацию → создаётся **generation job** на бэкенде.
4. Job проходит `running → succeeded`; **файл сохранён в object storage**, доступен по подписанному URL.
5. Создаётся **Asset** с рецептом, привязанный к проекту и модели.
6. **Кредиты списаны** через ledger на старте job; при ошибке job — **возвращены**.
7. Пользователь **выходит и входит снова** — проект, Canvas-граф и ассеты **восстанавливаются с сервера** (не из localStorage).
8. Ошибки генерации не приводят к потере кредитов и не оставляют «висящих» ассетов.

Когда все 8 пунктов выполняются на реальном бэкенде — шаг 1 закрыт, можно расширять scope (P1: сущности, базовые агенты, история, YooKassa, профиль, аналитика).
